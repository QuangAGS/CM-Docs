# **05\_EGAL\_25.x\_Architecture\_Doctrine\_Revision.gdoc**

## **PART I — Architectural Pivot (Post Phase 6.1)**

### **1\. Executive Summary**

After completing:

```
PHASE 5 — Unified Communication Backbone
PHASE 6.1 — Communication Propagation
```

the project entered a critical architectural reassessment.

Originally, EGAL-25.x envisioned a fully unified communication model across:

```
EMAIL
IN_APP
WEB_PUSH
ZALO
TELEGRAM
WHATSAPP
```

where every supported channel was treated as a first-class notification transport.

However, after business-process analysis, elderly-user UX review, onboarding feasibility assessment, and operational cost analysis, the doctrine was revised.

The revised doctrine recognizes an important distinction:

```
System-owned communication
≠
Community social interaction
```

This document formalizes the new architecture.

---

## **2\. Original Assumption (Deprecated)**

The previous assumption was:

```
User chooses preferred channel
→ System routes notification
→ Adapter delivers
```

Example:

```
preferred = ZALO
→ system attempts Zalo delivery

preferred = Telegram
→ system attempts Telegram delivery
```

This model assumed:

```
social channel
=
official communication transport
```

During Phase 6.1, multiple problems emerged.

### **Problem 1 — Elder User Onboarding Complexity**

Many elderly users:

```
do not use email actively
do not understand bots
do not understand QR onboarding
may only know one familiar app
```

Examples:

```
Zalo only
Telegram only
WhatsApp because children live abroad
```

This makes channel onboarding difficult and fragile.

---

### **Problem 2 — Admin Operational Burden**

Using Personal Accounts (PA) such as:

```
Zalo Personal (PZ)
Telegram Personal
WhatsApp Personal
```

as canonical communication transport creates:

```
manual onboarding dependency
manual delivery dependency
human workload escalation
```

Result:

```
CLAN_ADMIN becomes overloaded.
```

This violates scalability goals.

---

### **Problem 3 — Third-party Dependency Risk**

Relying on external social platforms for canonical delivery introduces:

```
API restrictions
rate limits
policy changes
privacy uncertainty
vendor lock-in
service instability
```

Especially problematic:

```
Zalo Personal
≠
official automation platform
```

The backend cannot reliably automate Personal Zalo communication.

---

### **Problem 4 — Notification Reliability**

Canonical business communication requires:

```
predictability
auditability
delivery traceability
stable routing
```

Social platforms are better suited for:

```
human discussion
community interaction
informal clan communication
```

rather than authoritative business notifications.

---

## **3\. Architectural Pivot**

The project therefore pivots from:

```
Unified Multi-Channel Communication
```

to:

```
System-Owned Communication First
```

Meaning:

```
Official business communication
must prioritize
system-owned channels.
```

while:

```
social platforms
remain optional extensions.
```

---

## **4\. Revised Communication Layer Model**

EGAL-25.x now separates communication into two distinct tiers.

### **Tier 1 — Canonical Communication Layer**

Purpose:

```
official business communication
```

Channels:

```
EMAIL
IN_APP
WEB_PUSH
```

Characteristics:

```
system-owned
automatable
auditable
policy-driven
low-cost
reliable
```

Examples:

```
approval notice
password reset
security alert
account status
clan announcements
reminders
system events
```

These channels are:

```
official
canonical
authoritative
```

---

### **Tier 2 — Community Social Layer**

Purpose:

```
human interaction
community discussion
social belonging
```

Channels:

```
ZALO GROUP
TELEGRAM GROUP
WHATSAPP GROUP
```

Characteristics:

```
optional
clan-level
human-centric
informal
non-canonical
```

Examples:

```
daily chat
family photos
casual discussion
announcements by clan members
event coordination
```

These channels are:

```
community spaces
```

NOT:

```
official notification transport
```

---

## **5\. New Strategic Principle**

The revised architecture follows:

```
WAC (Web App Clan)
must remain operationally independent
from social networks.
```

Meaning:

Even if:

```
Zalo fails
Telegram changes policy
WhatsApp becomes unavailable
```

the system still operates fully using:

```
EMAIL
IN_APP
WEB_PUSH
```

Business continuity remains intact.

This is a critical long-term stability principle.

---

## **6\. Architectural Outcome**

The resulting architecture becomes:

```
Business Event
→ NotificationOrchestrator
→ Policy Resolution
→ Persistence
→ Delivery

Tier 1:
EMAIL / IN_APP / WEB_PUSH

Tier 2:
Social Groups (independent)
```

Result:

```
cleaner architecture
lower maintenance cost
less vendor dependency
better reliability
better scalability
elder-friendly UX
```

---
