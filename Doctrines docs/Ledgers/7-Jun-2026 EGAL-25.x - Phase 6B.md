# **1\. Mục tiêu của Phase 6B**

Phase 6B nhằm mở rộng Communication Core của EGAL bằng cách bổ sung các business communication events theo mô hình silent emit.

Phase này KHÔNG nhằm:

* redesign authentication  
* redesign security recovery  
* redesign approval workflow  
* bật delivery thật  
* thay đổi UI/UX hiện tại  
* thay đổi business flow hiện tại

Mục tiêu duy nhất:

* chuẩn hóa event emission  
* persist communication history  
* xây dựng audit trail  
* chuẩn bị infrastructure cho delivery tương lai

# **2\. Core Doctrine**

## **2.1 Silent Emit Doctrine**

Tất cả communication events trong Phase 6B phải:

```
executeImmediately = false
```

Nghĩa là:

* event được tạo  
* event được persist  
* event được audit  
* event KHÔNG được gửi

---

## **2.2 Separation of Concerns**

Communication Core KHÔNG chịu trách nhiệm:

* xác minh danh tính  
* password recovery security  
* device trust  
* passkey recovery  
* anti-fraud decision  
* authentication enforcement

Communication Core chỉ trả lời:

```
"Điều gì đã xảy ra?"
```

Security System mới trả lời:

```
"Làm sao xác minh đúng người?"
```

---

## **2.3 Delivery Independence**

Communication events phải độc lập với:

* email provider  
* web push provider  
* SMS provider  
* UI notification  
* TTS  
* in-app rendering

Delivery sẽ được triển khai ở Phase sau.

# **3\. Users Status Doctrine**

## **3.1 Canonical enum\_users\_status**

```
enum_users_status = {
  CHO_DUYET,
  DA_DUYET,
  BI_KHOA,
  BI_CAM,
  TAM_NGUNG,
  TU_CHOI
}
```

---

## **3.2 Semantic Definition**

### **CHO\_DUYET**

User đã đăng ký nhưng chưa được duyệt.

---

### **DA\_DUYET**

User đã được approve và active.

---

### **TU\_CHOI**

User bị reject trong approval workflow.

Đây là business governance state.

KHÔNG phải security state.

---

### **TAM\_NGUNG**

Account tạm ngưng vận hành/nghiệp vụ.

Ví dụ:

* compliance hold  
* missing documents  
* operational suspension

---

### **BI\_KHOA**

Temporary security lock.

Ví dụ:

* brute-force protection  
* suspicious login  
* anti-bot  
* rate limit

Đây là temporary security enforcement state.

---

### **BI\_CAM**

Permanent enforcement/moderation ban.

Ví dụ:

* abuse  
* fraud  
* severe policy violation

Đây là terminal enforcement state.

---

# **4\. Security Lock Doctrine**

## **4.1 Overlay Security Model**

BI\_KHOA là temporary overlay state.

Doctrine chuẩn:

```
previous_status -> BI_KHOA
BI_KHOA -> previous_status
```

Ví dụ:

```
DA_DUYET -> BI_KHOA -> DA_DUYET
```

hoặc:

```
TU_CHOI -> BI_KHOA -> TU_CHOI
```

---

## **4.2 Required Support Fields**

Khuyến nghị:

```
locked_previous_status
status_changed_at
```
