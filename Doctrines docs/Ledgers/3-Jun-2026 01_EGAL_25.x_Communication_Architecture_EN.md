# **01\_EGAL\_25.x\_Communication\_Architecture\_ADR[^1].md**

# **PART I — CONTEXT & BACKGROUND**

## **1.1 Historical Context**

### **From EGAL-24.x.x → EGAL-25.x.x**

The evolution toward **EGAL-25.x.x** begins after the successful stabilization of:

```
EGAL-24.x.x
Elder Guidance & Accessibility Layer
```

for the myClan genealogy platform.

EGAL-24 successfully established:

* elder-friendly onboarding  
* low-anxiety interaction model  
* voice-supported accessibility  
* submit-gated validation  
* controlled attention routing  
* browser-safe voice behavior  
* accessibility-first UX for elderly users

The fundamental doctrine of EGAL-24 can be summarized as:

```
Technology must adapt to elderly users,
not the opposite.
```

This architectural direction proved successful during UAT across:

* LoginForm  
* RegisterForm  
* JoinClanForm  
* CreateClanForm  
* ForgotPasswordForm  
* VerifyResetCodeForm  
* ChangePasswordForm  
* WaitingPage

As a result:

```
EGAL-24 Elder Support Doctrine
is considered production-safe.
```

However, a new problem emerged.

The platform achieved:

```
elder-friendly onboarding
```

but communication between:

```
system ↔ user
```

remained heavily dependent on:

```
email only
```

This creates significant usability friction.

---

## **1.2 Business Reality of myClan**

The myClan platform serves a unique user demographic.

Unlike conventional productivity platforms, myClan operates around:

```
family genealogy
clan management
multi-generation participation
```

This means the platform simultaneously serves:

### **Elderly Vietnamese users**

Characteristics:

* often unfamiliar with email  
* rarely check inboxes  
* strongly prefer Zalo  
* lower tolerance for technical complexity  
* high anxiety toward unfamiliar UI

---

### **Younger Vietnamese users**

Characteristics:

* comfortable with Telegram  
* comfortable with email  
* open to web notifications  
* more technically familiar

---

### **International users**

Characteristics:

* WhatsApp-heavy  
* email-oriented  
* cross-country communication needs

---

### **Clan Administrators**

Characteristics:

* responsible for onboarding  
* approval workflow owners  
* account recovery assistance  
* communication coordinators

This diversity creates an unavoidable architectural challenge:

```
one communication channel
cannot satisfy all users.
```

---

## **1.3 Limitation of Current Architecture**

The existing EGAL-24 communication model assumes:

```
email = communication
```

This assumption is insufficient.

Problems observed:

### **Problem A — Elderly users ignore email**

For many elderly Vietnamese users:

```
email exists only for registration.
```

In practice:

* inbox is rarely checked  
* password recovery becomes difficult  
* approval notifications are missed  
* onboarding confidence decreases

This directly contradicts:

```
Elder Support Doctrine
```

because:

```
supportive onboarding
without supportive communication
remains incomplete.
```

---

### **Problem B — Communication friction**

Current business flows require communication events such as:

```
approval notice
password reset
support requests
status update
invitation
review reminder
```

Under email-only architecture:

many users:

```
do not receive information
at the right time
through preferred channels.
```

---

### **Problem C — Overdependence on a single medium**

Email-only architecture introduces:

```
single-point communication dependency
```

Meaning:

If users do not engage with email:

```
system-user relationship weakens.
```

This becomes especially problematic for:

```
elderly clan members
```

whose primary digital communication habit is:

```
Zalo
```

rather than email.

---

### **Problem D — International scalability issue**

myClan is intended to support:

```
Vietnamese users
+
international family members
```

A single communication strategy:

```
email only
```

cannot adequately support:

```
Telegram-oriented users
WhatsApp-oriented users
```

across countries.

---

## **1.4 Strategic Constraint**

The solution space is constrained by several realities.

### **Constraint \#1 — Free-first principle**

Recurring operational costs must remain low.

The architecture should prioritize:

```
free
or near-free communication channels
```

before paid services.

This excludes:

```
SMS-first architecture
```

despite SMS being technically reliable.

SMS may remain:

```
future optional layer
```

but SHALL NOT become:

```
core dependency
```

for EGAL-25.

---

### **Constraint \#2 — No Zalo OA dependency**

The project intentionally avoids:

```
Zalo Official Account (OA)
```

Reasons:

* operational complexity  
* policy dependency  
* platform governance risk  
* long-term vendor coupling

Instead:

the system assumes:

```
Personal Zalo (PZ)
```

for both:

```
Clan Admin
+
end user
```

This creates limitations:

because Personal Zalo lacks:

```
stable official automation API
```

---

### **Constraint \#3 — Elderly-friendly onboarding**

The communication solution must preserve:

```
low anxiety
low friction
```

The platform SHALL NOT require:

```
complex social account binding
mandatory bot setup
multi-step technical onboarding
```

during registration.

The onboarding philosophy remains:

```
simple first
optional later
```

---

### **Constraint \#4 — Security preservation**

Communication flexibility SHALL NOT compromise:

```
identity assurance
account recovery
auditability
```

Higher privilege users:

```
USER
EDITOR
ADMIN
```

still require:

```
reliable official communication
```

This strongly favors:

```
email
```

as a mandatory channel.

---

## **1.5 Emerging Architectural Insight**

During exploratory dialogue and business analysis, a major realization emerged.

The original assumption was:

```
social channel
=
communication integration
```

This led toward:

```
social API coupling
```

However, this direction proved:

```
fragile
costly
platform-dependent
elder-unfriendly
```

A new architectural insight replaced this assumption:

```
social platforms
should not become system dependencies.
```

Instead:

they should become:

```
communication preferences
```

owned by users.

This insight becomes the foundation of:

```
EGAL-25.x.x
Unified Communication Architecture
```

---

# **PART II — PROBLEM STATEMENT**

## **2.1 Core Problem**

The current architecture lacks:

```
multi-channel communication orchestration
```

while serving:

```
heterogeneous user groups
```

with radically different:

```
technology habits
communication preferences
accessibility needs
```

This creates a mismatch between:

```
system communication behavior
```

and:

```
real-world user behavior.
```

---

## **2.2 Architectural Problem Categories**

### **Problem Category A**

## **Communication Is Hardcoded**

Current architecture effectively assumes:

```
Email = communication system
```

This hardcoding causes:

* poor elderly adoption  
* missed notifications  
* low engagement  
* fragile onboarding continuity

The system lacks:

```
channel abstraction
```

and:

```
communication flexibility.
```

---

### **Problem Category B**

## **Elderly Communication Gap**

EGAL-24 successfully solved:

```
elder interaction
```

but not:

```
elder communication.
```

This creates an architectural inconsistency.

The system speaks:

```
elder language
```

during onboarding,

but reverts to:

```
technical communication assumptions
```

after onboarding.

This contradiction weakens:

```
elder confidence
```

in the platform.

---

### **Problem Category C**

## **Social Platform Dependency Risk**

An obvious solution would be:

```
integrate Zalo
integrate Telegram
integrate WhatsApp
```

as:

```
identity providers
```

or:

```
full communication APIs.
```

However this introduces:

### **vendor lock-in**

### **API instability**

### **platform governance dependency**

### **onboarding complexity**

### **security fragility**

Especially problematic:

```
Personal Zalo
```

because:

official automation support is limited.

This makes:

```
social-as-core-infrastructure
```

a poor long-term architectural choice.

---

### **Problem Category D**

## **Missing Notification Governance**

Business events currently lack:

```
notification governance
```

There is no centralized logic deciding:

```
who gets notified
through which channel
at what reliability level
with what priority
```

This creates:

```
scattered communication behavior
```

across modules.

Examples:

```
RegisterForm
Approval workflow
Password reset
Security warning
Role change
```

currently lack a unified notification doctrine.

---

### **Problem Category E**

## **Role vs Accessibility Confusion**

A misleading simplification initially emerged:

```
elderly users = VIEWER role
```

Although practical in many cases:

this assumption is architecturally incorrect.

Possible real-world cases:

```
elder clan leader
elder historian
elder editor
```

This means:

```
elder support
```

cannot depend on:

```
authorization role.
```

Accessibility needs must remain:

```
independent
```

from:

```
permission structure.
```

---

## **2.3 Design Requirements**

EGAL-25 therefore requires:

### **Requirement \#1**

Support:

```
multiple communication channels
```

without:

```
tight platform coupling.
```

---

### **Requirement \#2**

Preserve:

```
elder-friendly simplicity.
```

No forced technical setup.

---

### **Requirement \#3**

Support:

```
free-first operation.
```

---

### **Requirement \#4**

Maintain:

```
official communication reliability
```

for privileged users.

---

### **Requirement \#5**

Introduce:

```
notification orchestration
```

as centralized infrastructure.

---

### **Requirement \#6**

Preserve:

```
EGAL-24 doctrines
```

especially:

```
low anxiety
low friction
controlled complexity
```

---

# **PART III — ARCHITECTURAL DECISION**

## **3.1 Final Decision**

EGAL-25 SHALL introduce:

```
Unified Communication Layer (UCL)
```

based on:

```
communication preference
```

rather than:

```
social platform integration
```

The architecture SHALL separate:

```
authentication
authorization
communication
notification delivery
```

into independent layers.

---

## **3.2 Foundational Decision**

The system SHALL NOT treat:

```
social applications
```

as:

```
identity providers
```

Instead:

social platforms SHALL be treated as:

```
communication preference channels
```

only.

This is the most important architectural decision of EGAL-25.

Core philosophy:

```
Identity is NOT communication.

Communication is NOT authentication.
```

---

## **3.3 Communication Model**

The system SHALL support:

```
multi-channel communication
```

using:

```
adapter architecture
```

Supported channels:

```
IN_APP
EMAIL
ZALO_PZ
TELEGRAM
WHATSAPP
WEB_PUSH
```

Each channel SHALL act as:

```
delivery adapter
```

instead of:

```
business logic owner.
```

Meaning:

business flows SHALL NOT directly call:

```
Telegram
Email
Zalo
```

Instead:

business flows emit:

```
notification intent
```

which becomes centrally orchestrated.

Example:

```
USER_APPROVED
```

↓

```
Notification Policy Engine
```

↓

```
select channels
```

↓

```
deliver
```

---

## **3.4 Zalo Decision**

The project explicitly rejects:

```
Zalo OA dependency
```

and:

```
unofficial Personal Zalo automation APIs
```

for production core.

Instead:

Personal Zalo SHALL become:

```
Manual-Assisted Communication Channel
```

Meaning:

the platform SHALL support:

* QR connection  
* guided onboarding  
* deep link assistance  
* pre-generated messages  
* copy-to-send workflow  
* manual notification queue

But SHALL NOT rely on:

```
unofficial bot automation
```

as production infrastructure.

---

## **3.5 Deferred Communication Binding Decision**

Users SHALL NOT be forced to:

```
bind communication channels
```

during registration.

Instead:

the system stores:

```
communication preference
```

first.

Binding occurs:

```
later
```

if needed.

This doctrine minimizes:

```
elder anxiety
onboarding friction
```

while preserving:

```
communication flexibility.
```

```
EGAL-25.x.x
```

# **4.2 PRINCIPLE \#1**

# **Identity is NOT Communication**

## **Decision**

Authentication identity SHALL remain independent from communication systems.

Meaning:

```
who the user is
```

must remain separate from:

```
how the system communicates
```

with that user.

---

## **Rationale**

Many systems mistakenly couple:

```
identity
```

with:

```
social platform accounts
```

Examples:

```
Login with Facebook
Login with Telegram
Login with WhatsApp
```

This introduces:

* vendor lock-in  
* API fragility  
* onboarding complexity  
* migration difficulty  
* platform governance dependency

Such dependency is unacceptable for:

```
long-lived genealogy systems
```

because:

family platforms must survive:

```
decades
```

rather than:

```
social platform trends.
```

---

## **EGAL-25 Decision**

myClan SHALL authenticate users using:

```
phone
email
password
future MFA
```

NOT:

```
social identity.
```

Examples:

### **Allowed**

```
phone + password
email + password
```

---

### **Forbidden**

```
login via Zalo
login via Telegram
login via WhatsApp
```

unless explicitly approved in:

```
future ADR.
```

---

## **Doctrine Statement**

```
Identity must survive
outside communication platforms.
```

---

# **4.3 PRINCIPLE \#2**

# **Communication is NOT Authentication**

## **Decision**

Communication channels SHALL NOT become:

```
security dependencies.
```

Social platforms exist for:

```
communication convenience
```

NOT:

```
identity assurance.
```

---

## **Rationale**

Communication platforms are:

```
unstable
```

from a system-design perspective.

Users may:

* uninstall apps  
* change phone numbers  
* stop using Telegram  
* abandon WhatsApp  
* lose Zalo access

Yet:

```
system identity
```

must remain stable.

If communication becomes authentication:

the system becomes:

```
fragile.
```

---

## **Example**

### **Wrong Design**

```
User must connect Zalo
before account activation.
```

Problem:

If Zalo changes:

```
account access breaks.
```

---

### **Correct Design**

```
User chooses Zalo
as preferred communication.
```

But:

```
account remains independent.
```

---

## **Doctrine Statement**

```
Communication may change.

Identity must remain stable.
```

---

# **4.4 PRINCIPLE \#3**

# **In-App Notification Is Canonical**

## **Decision**

Every notification SHALL first exist:

```
inside myClan
```

before:

```
external delivery.
```

This means:

```
in-app notification
```

becomes:

```
source of truth.
```

---

## **Rationale**

External channels can fail.

Examples:

```
email not opened
Telegram unavailable
Zalo not connected
WhatsApp inactive
push disabled
```

However:

the platform must preserve:

```
communication auditability.
```

Thus:

all business communication SHALL first be:

```
stored
```

in system database.

---

## **Canonical Flow**

Architecture:

```
Business Event
    ↓
Create Notification Record
    ↓
External Delivery
```

NOT:

```
Business Event
    ↓
Send Email Directly
```

---

## **Benefits**

### **Reliability**

Message history survives.

---

### **Auditability**

Admin can inspect:

```
who was notified
when
through which channel
delivery result
```

---

### **Recovery**

Failed delivery can be retried.

---

### **Multi-channel flexibility**

One notification:

```
many channels.
```

---

## **Doctrine Statement**

```
External channels deliver.

myClan remembers.
```

---

# **PART V — USER SEGMENTATION MODEL**

## **5.1 Purpose of User Segmentation**

The communication architecture of:

```
EGAL-25.x.x
```

must acknowledge a fundamental reality:

```
not all users communicate the same way.
```

Unlike generic SaaS systems, myClan serves:

```
multi-generational
multi-cultural
multi-technology users.
```

This creates:

```
communication diversity.
```

A single communication strategy:

```
cannot adequately support
all user groups.
```

Therefore:

EGAL-25 introduces:

```
User Segmentation Model
```

to guide:

* onboarding  
* communication routing  
* UI assistance  
* notification behavior  
* future automation decisions

This segmentation is:

```
behavioral
```

NOT:

```
security-oriented.
```

Meaning:

segmentation SHALL NOT replace:

```
authorization roles.
```

Instead:

it complements:

```
communication experience.
```

---

# **5.2 Foundational Principle**

EGAL-25 explicitly rejects:

```
one-size-fits-all communication.
```

Instead:

communication SHALL become:

```
segment-aware.
```

This means:

the system SHOULD adapt to:

```
user communication habits
```

rather than forcing users to adapt to:

```
technical assumptions.
```

This philosophy directly extends:

```
EGAL-24 Elder Support Doctrine
```

into:

```
Communication Experience Design.
```

# **11.2 Rejected Alternative A**

# **Social Platforms as Identity Providers**

## **Rejected Proposal**

```

WhatsApp login
```

as:

```
authentication identity.
```

Equivalent model:

```
social account
=
myClan account
```

---

## **Why It Was Considered**

Advantages initially appeared attractive:

* faster onboarding  
* less password management  
* easier login experience  
* familiar user habit

Especially:

```
elder users
```

may appear more comfortable with:

```

```

than:

```

```

---

## **Why It Was Rejected**

### **Problem 1 — Vendor Lock-in**

```
third-party platform survival.
```

Example:

If:

```
Zalo policy changes
Telegram account lost
WhatsApp blocked
```

then:

```
account access breaks.
```

This is unacceptable for:

```
long-lived genealogy systems.
```

---

### **Problem 2 — Platform Dependency**

myClan becomes:

```
externally controlled.
```

Future architectural freedom decreases.

---

### **Problem 3 — Security Fragility**

Authentication assurance becomes:

```
outside system control.
```

Especially dangerous for:

```
EDITOR
ADMIN
```

roles.

---

### **Problem 4 — Migration Difficulty**

Changing communication platform later becomes:

```
identity migration problem.
```

Instead of:

```
simple preference update.
```

---

## **Final Decision**

Rejected.

Replaced by:

```
Identity ≠ Communication
```

---

## **Doctrine Statement**

```
Social platforms may change.

Identity must survive.
```
