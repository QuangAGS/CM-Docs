**TÀI LIỆU THIẾT KẾ CHÍNH THỨC**  
 **EGAL-25.x — ONBOARDING PROCESS DOCTRINE**

Phiên bản: 1.0.0  
 Ngày: 24/06/2026

# **I. Mục tiêu kiến trúc**

Onboarding là một Business Process kéo dài, không chỉ là điền hồ sơ. Chuỗi gồm: hoàn thiện thông tin cá nhân; tạo Member dự bị; khai báo nhánh gia đình; gửi yêu cầu nhập tộc; thẩm định; ghép cây; chuyển Member từ DU\_BI sang CHINH\_THUC; nâng quyền User. Vì vậy cần onboarding\_cases làm Business Case Aggregate Root.

# **II. Các thực thể chính**

users: tài khoản/xác thực/quyền truy cập.

members: thực thể gia phả.

branches: nhánh cây phả hệ.

onboarding\_cases: hồ sơ onboarding và workflow.

business\_process\_logs: Business Process Ledger.

notifications, notification\_recipients, notification\_deliveries: Communication Ledger.

# **III. Phân lớp trạng thái**

users.status phục vụ an ninh: CHO\_DUYET, DA\_DUYET, BI\_KHOA, BI\_CAM, TAM\_NGUNG, TU\_CHOI.

members.status phục vụ gia phả: DU\_BI, CHINH\_THUC.

branches.status phục vụ nhánh: DRAFT, PROVISIONAL, SUBMITTED, UNDER\_REVIEW, APPROVED, MERGED, REJECTED, ARCHIVED.

onboarding\_cases.status phục vụ workflow: DRAFT, PROFILE\_COMPLETED, FAMILY\_TREE\_DRAFT, SUBMITTED, UNDER\_REVIEW, NEEDS\_REVISION, APPROVED, REJECTED, MERGED, CANCELLED, EXPIRED.

# **IV. Branch dự bị**

User mới chỉ được tạo Provisional/Unlinked Branch vì chưa biết vị trí chính xác trong cây tổng. CLAN\_ADMIN hoặc người có thẩm quyền quyết định ghép. Quan hệ mục tiêu: onboarding\_cases 1:N branches 1:N members. Phase 6B có thể triển khai tối giản một case có một primary\_branch.

# **V. Schema onboarding\_cases**

Các trường cốt lõi: id, correlation\_id, case\_type, status, user\_id, tenant\_id, primary\_member\_id, primary\_branch\_id, submitted\_at, reviewed\_at, reviewed\_by, approved\_at, rejected\_at, merged\_at, review\_note, rejection\_reason, revision\_request, metadata, created\_at, updated\_at, deleted\_at.

Index tối thiểu: user\_id, tenant\_id, status, case\_type, correlation\_id, submitted\_at, reviewed\_by, created\_at DESC.

# **VI. Metadata doctrine**

Không dùng metadata thay cho cột cần query/filter/join/phân quyền. status, user\_id, tenant\_id, reviewed\_by, submitted\_at, primary\_member\_id, primary\_branch\_id là cột riêng. Metadata chỉ lưu dữ liệu linh hoạt: mức hoàn thiện profile, số đời, số member, nguồn submit, IP.

# **VII. Business Process Ledger**

Process types đề xuất: ONBOARDING\_CASE\_CREATE, ONBOARDING\_PROFILE\_SAVE, ONBOARDING\_PROFILE\_COMPLETE, ONBOARDING\_BRANCH\_CREATE, ONBOARDING\_BRANCH\_UPDATE, ONBOARDING\_SUBMIT, ONBOARDING\_REVIEW\_START, ONBOARDING\_REVISION\_REQUEST, ONBOARDING\_APPROVE, ONBOARDING\_REJECT, ONBOARDING\_BRANCH\_MERGE, ONBOARDING\_COMPLETE, CLAN\_ONBOARDING\_CREATE, CLAN\_ONBOARDING\_CONFIGURE, CLAN\_TREE\_INITIALIZE, CLAN\_ONBOARDING\_ACTIVATE.

# **VIII. Communication Ledger**

Không cần communication\_events riêng. notifications \+ notification\_recipients \+ notification\_deliveries đã là communication ledger. notifications nên có correlation\_id và tenant\_id nullable, với index correlation\_id và tenant\_id. Event onboarding: ONBOARDING\_STARTED, ONBOARDING\_PROFILE\_COMPLETED, ONBOARDING\_SUBMITTED, ONBOARDING\_UNDER\_REVIEW, ONBOARDING\_REVISION\_REQUESTED, ONBOARDING\_APPROVED, ONBOARDING\_REJECTED, ONBOARDING\_BRANCH\_MERGED, ONBOARDING\_COMPLETED.

# **IX. Data flow**

Bắt đầu: create onboarding\_cases → business\_process\_logs → notification tùy chọn.

Hoàn thành profile: update contacts/addresses/members/users.member\_id/case → log.

Tạo nhánh: create branch và provisional members → update primary\_branch\_id → log.

Submit: case/branch SUBMITTED → log → thông báo admin.

Revision: NEEDS\_REVISION \+ revision\_request → log → thông báo user.

Approve/Merge: transaction cập nhật case, branch, members, users; ghi business log, audit log, notification.

# **X. Correlation ID, transaction và guard**

Một correlation\_id xuyên suốt onboarding\_cases, business\_process\_logs, audit\_logs, notifications. Thay đổi trạng thái chính thức chạy transaction. notification\_deliveries có thể async sau transaction. Middleware xét users.status, users.role, users.member\_id, onboarding\_cases.status, tenant.status.

# **XI. Kết luận**

users.status \= an ninh; members.status \= gia phả; branches.status \= nhánh; onboarding\_cases.status \= workflow; business\_process\_logs \= lịch sử thực thi; audit\_logs \= lịch sử biến động dữ liệu; notifications \= lịch sử truyền thông. Kiến trúc hỗ trợ review nhiều vòng, merge, retry, audit và mở rộng BP.

**XII. Refinement Recommendations — Data Lifecycle, Concurrency & Contract Integrity**

## **XII.1. Vòng đời dữ liệu khi hồ sơ bị từ chối hoặc hủy**

Khuyến nghị A là đúng.

Khi `onboarding_cases.status` chuyển sang `REJECTED` hoặc `CANCELLED`, không nên xóa vật lý `branches` hoặc `members` dự bị. Các dữ liệu này là bằng chứng nghiệp vụ, có thể cần đối chiếu khi người dùng nộp lại hồ sơ, khiếu nại, hoặc khi quản trị viên rà soát lịch sử.

### **Quy tắc đề xuất**

| Sự kiện | onboarding\_cases | branches | members |
| ----- | ----- | ----- | ----- |
| User tự hủy hồ sơ | `CANCELLED` | `ARCHIVED` | Giữ `DU_BI` |
| Admin từ chối hồ sơ | `REJECTED` | `REJECTED` hoặc `ARCHIVED` | Giữ `DU_BI` |
| Hồ sơ hết hạn | `EXPIRED` | `ARCHIVED` | Giữ `DU_BI` |
| User nộp lại hồ sơ | Tạo `onboarding_cases` mới | Tạo branch mới hoặc tái sử dụng branch cũ theo chính sách | Có thể tái sử dụng Member dự bị |

### **Quy tắc quan trọng**

* Không dùng `deleted_at` để biểu diễn trạng thái nghiệp vụ như `REJECTED`, `CANCELLED`, `ARCHIVED`.  
* `deleted_at` chỉ nên dùng cho soft-delete mang tính kỹ thuật hoặc quản trị dữ liệu.  
* UI cây phả hệ chính thức chỉ hiển thị:  
  * `members.status = CHINH_THUC`  
  * branch đã `MERGED`  
* Branch `REJECTED`, `ARCHIVED`, `CANCELLED` chỉ xuất hiện trong màn hình quản trị hoặc lịch sử onboarding.

### **Bổ sung vào Business Process Ledger**

Các process type nên có:

```
ONBOARDING_CANCEL
ONBOARDING_REJECT
ONBOARDING_CASE_EXPIRE
ONBOARDING_BRANCH_ARCHIVE
ONBOARDING_CASE_REOPEN
```

Điểm cần tinh chỉnh: `ONBOARDING_CASE_REOPEN` chỉ dùng khi hệ thống thật sự cho phép mở lại **cùng một case**. Nếu chính sách là “nộp lại \= hồ sơ mới”, nên dùng:

```
ONBOARDING_CASE_CREATE
```

và metadata lưu:

```json
{
  "previous_case_id": "uuid-cua-case-cu",
  "reapplication_reason": "User submitted updated family information"
}
```

## **XII.2. Khóa chỉnh sửa khi hồ sơ vào giai đoạn review hoặc merge**

Khuyến nghị B là rất quan trọng và nên đưa vào doctrine như một nguyên tắc bắt buộc.

Tuy nhiên, chỉ khóa ở Frontend là chưa đủ. Frontend có thể bị bỏ qua bằng API call trực tiếp, tab cũ, hoặc request gửi chậm.

Cần có **hai lớp khóa**:

### **Lớp 1 — UX Lock**

Khi:

```
onboarding_cases.status IN (
  SUBMITTED,
  UNDER_REVIEW,
  APPROVED,
  MERGED,
  REJECTED,
  CANCELLED
)
```

Frontend phải đưa branch sang chế độ read-only.

User không được:

* thêm Member dự bị;  
* sửa quan hệ cha/mẹ/vợ/chồng;  
* xóa Member;  
* thay đổi cấu trúc branch;  
* gửi lại hồ sơ khi chưa được yêu cầu chỉnh sửa.

Chỉ khi case ở:

```
DRAFT
PROFILE_COMPLETED
FAMILY_TREE_DRAFT
NEEDS_REVISION
```

thì user mới được chỉnh sửa.

### **Lớp 2 — Backend State Guard**

Mọi API mutation liên quan đến provisional branch phải kiểm tra:

```
onboarding_cases.status IN (
  DRAFT,
  PROFILE_COMPLETED,
  FAMILY_TREE_DRAFT,
  NEEDS_REVISION
)
```

Nếu không thỏa điều kiện, backend trả về lỗi nghiệp vụ, ví dụ:

```
ONBOARDING_CASE_NOT_EDITABLE
```

Không nên chỉ trả về lỗi chung kiểu `403 Forbidden`, vì Frontend cần biết đây là lỗi workflow để hiển thị đúng thông báo.

### **Khóa khi merge**

Khi Admin bắt đầu merge, cần chuyển case sang trạng thái trung gian:

```
MERGING
```

Luồng đề xuất:

```
UNDER_REVIEW
→ APPROVED
→ MERGING
→ MERGED
```

Lý do:

* `APPROVED` nghĩa là hồ sơ được chấp thuận về nghiệp vụ.  
* `MERGING` nghĩa là hệ thống đang thực hiện thao tác cấu trúc cây phả hệ.  
* `MERGED` nghĩa là transaction ghép cây đã hoàn tất thành công.

Nếu transaction merge thất bại:

```
MERGING → APPROVED
```

hoặc:

```
MERGING → MERGE_FAILED
```

Tôi nghiêng về phương án có trạng thái riêng:

```
MERGE_FAILED
```

vì nó giúp Admin biết hồ sơ đã được duyệt nhưng thao tác ghép cây chưa hoàn thành.

Enum `onboarding_case_status` nên mở rộng:

```
DRAFT
PROFILE_COMPLETED
FAMILY_TREE_DRAFT
SUBMITTED
UNDER_REVIEW
NEEDS_REVISION
APPROVED
MERGING
MERGED
MERGE_FAILED
REJECTED
CANCELLED
EXPIRED
```

## **XII.3. Composite Index cho Dashboard đa tenant**

Khuyến nghị C là đúng và nên bổ sung.

```
@@index([tenant_id, status], map: "idx_onboarding_tenant_status")
```

Đây là index cần thiết cho truy vấn phổ biến:

```
Lấy các hồ sơ onboarding đang chờ xử lý của tenant hiện tại.
```

Ví dụ:

```javascript
prisma.onboarding_cases.findMany({
  where: {
    tenant_id,
    status: {
      in: ['SUBMITTED', 'UNDER_REVIEW', 'NEEDS_REVISION']
    }
  },
  orderBy: {
    submitted_at: 'asc'
  }
});
```

Tuy nhiên, để phục vụ đúng query trên, index tốt hơn là:

```
@@index(
  [tenant_id, status, submitted_at(sort: Asc)],
  map: "idx_onboarding_tenant_status_submitted"
)
```

Index này hỗ trợ đồng thời:

1. lọc theo tenant;  
2. lọc theo trạng thái;  
3. sắp xếp danh sách hồ sơ theo thời điểm nộp.

Ngoài ra, nên có:

```
@@index(
  [user_id, status],
  map: "idx_onboarding_user_status"
)
```

Index này phục vụ State Guard và trang “Hồ sơ onboarding của tôi”.

### **Bộ index đề xuất cuối cùng**

```
@@index([tenant_id, status, submitted_at(sort: Asc)], map: "idx_onboarding_tenant_status_submitted")
@@index([user_id, status], map: "idx_onboarding_user_status")
@@index([correlation_id], map: "idx_onboarding_correlation")
@@index([reviewed_by], map: "idx_onboarding_reviewer")
@@index([primary_branch_id], map: "idx_onboarding_primary_branch")
@@index([created_at(sort: Desc)], map: "idx_onboarding_created_at")
```

Không cần tạo đồng thời cả:

```
@@index([tenant_id])
@@index([status])
@@index([tenant_id, status])
@@index([tenant_id, status, submitted_at])
```

vì sẽ gây dư thừa index. Với Dashboard, composite index ba cột thường đã bao phủ tốt nhu cầu chính.

## **XII.4. Đồng bộ enum giữa Prisma schema, Business Ledger và validator**

Khuyến nghị D là bắt buộc.

Hiện doctrine có ba lớp phải luôn đồng bộ:

| Lớp | Vai trò |
| ----- | ----- |
| PostgreSQL enum `business_process_type` | Chặn dữ liệu process type không hợp lệ ở DB |
| Prisma enum `business_process_type` | Type safety trong application |
| `business-log-schemas.js` | Chuẩn hóa và validate metadata trước khi ghi log |

Một `process_type` chỉ được xem là hoàn chỉnh khi có đủ cả ba lớp.

Ví dụ, nếu schema DB có:

```
ONBOARDING_SUBMIT
```

nhưng `BusinessLogSchemas.ONBOARDING_SUBMIT` chưa tồn tại, thì `business-logger.service.js` phải từ chối ghi log. Đây là hành vi đúng vì tránh metadata không có cấu trúc.

Ví dụ validator nên bổ sung:

```javascript
ONBOARDING_SUBMIT: (payload) => {
  if (!payload.user_id) {
    throw new Error('ONBOARDING_SUBMIT requires user_id');
  }

  if (!payload.member_id) {
    throw new Error('ONBOARDING_SUBMIT requires member_id');
  }

  if (!payload.branch_id) {
    throw new Error('ONBOARDING_SUBMIT requires branch_id');
  }

  return {
    user_id: payload.user_id,
    member_id: payload.member_id,
    branch_id: payload.branch_id,
    member_count: Number.parseInt(payload.member_count, 10) || 0,
    generation_depth: Number.parseInt(payload.generation_depth, 10) || null,
    submitted_from: payload.submitted_from || 'WEB'
  };
},
```

Tuy nhiên, theo Snapshot Doctrine hiện tại, `user_id`, `member_id`, `branch_id` không nên chỉ nằm trong `payload`.

Chúng nên xuất hiện trong `context` của `business_process_logs`:

```javascript
context: {
  target_id: onboardingCase.id,
  target_name: `Onboarding case of ${user.full_name}`,
  user_id: user.id,
  member_id: member.id,
  branch_id: branch.id
}
```

Còn `payload` nên chỉ giữ thông tin biến động hoặc dữ liệu nghiệp vụ của lần thực thi:

```javascript
payload: {
  member_count: 8,
  generation_depth: 4,
  submitted_from: 'WEB'
}
```

Điều này giúp metadata nhất quán:

```json
{
  "context": {
    "target_id": "onboarding-case-uuid",
    "target_name": "Onboarding case of Nguyễn Văn A",
    "user_id": "user-uuid",
    "member_id": "member-uuid",
    "branch_id": "branch-uuid"
  },
  "payload": {
    "member_count": 8,
    "generation_depth": 4,
    "submitted_from": "WEB"
  }
}
```

# **Kết luận review**

Bốn phản biện đều có giá trị và nên được đưa vào tài liệu. Nhưng phiên bản chính thức nên bổ sung thêm ba quyết định kiến trúc:

1. Có trạng thái `MERGING` và `MERGE_FAILED` để tách “được duyệt” khỏi “đã ghép cây thành công”.  
2. State Guard phải được thực thi ở cả Frontend và Backend.  
3. Các ID định danh chính của đối tượng phải được đặt trong `context` snapshot; `payload` chỉ chứa dữ liệu đặc thù của lần thực thi.

Như vậy doctrine Onboarding sẽ đủ vững cho Phase 6B, đồng thời không bị giới hạn khi sau này triển khai merge phức tạp, retry, review nhiều vòng hoặc workflow cho nhiều cấp trưởng chi.

# **I. Mục tiêu kiến trúc**

Onboarding trong EGAL không chỉ là “điền thêm hồ sơ”.

Đây là một chuỗi Business Process có thể kéo dài nhiều ngày hoặc nhiều tuần, gồm:

1. Người dùng hoàn thiện thông tin cá nhân.  
2. Hệ thống tạo Member dự bị.  
3. Người dùng khai báo nhánh gia đình nhỏ.  
4. Người dùng gửi yêu cầu nhập tộc.  
5. Clan Admin hoặc trưởng chi thẩm định.  
6. Nhánh dự bị được ghép vào cây phả hệ chính thức.  
7. Các Member được chuyển từ `DU_BI` sang `CHINH_THUC`.  
8. User được nâng quyền từ `VIEWER` sang `USER` hoặc `EDITOR`.

Vì vậy, onboarding không nên chỉ được suy luận từ:

```
members.status
members.role
members.note
```

Các trường này chỉ phản ánh **trạng thái hiện tại của Member**, không phản ánh đầy đủ:

* Người dùng đang ở bước onboarding nào.  
* Hồ sơ đã gửi hay chưa.  
* Có đang chờ duyệt hay không.  
* Ai đang xử lý hồ sơ.  
* Nhánh gia đình dự bị nào thuộc về hồ sơ nào.  
* Admin yêu cầu bổ sung thông tin gì.  
* Hồ sơ đã bị từ chối bao nhiêu lần.  
* Khi nào user gửi lại hồ sơ.  
* Lịch sử ghép cây.  
* Các Business Process liên quan.  
* Các notification đã gửi.

Do đó, EGAL cần một thực thể riêng:

```
onboarding_cases
```

Đây là **Business Case Aggregate Root** của toàn bộ onboarding.

## **3\. `onboarding_cases.status`**

Dùng cho trạng thái workflow.

```
enum onboarding_case_status {
```

Ý nghĩa:

| Status | Ý nghĩa |

# **XV. Refinement Recommendations — Data Lifecycle, Concurrency & Contract Integrity**

## **XV.1. Vòng đời dữ liệu khi hồ sơ bị từ chối hoặc hủy**

Khuyến nghị A là đúng.

Khi `onboarding_cases.status` chuyển sang `REJECTED` hoặc `CANCELLED`, không nên xóa vật lý `branches` hoặc `members` dự bị. Các dữ liệu này là bằng chứng nghiệp vụ, có thể cần đối chiếu khi người dùng nộp lại hồ sơ, khiếu nại, hoặc khi quản trị viên rà soát lịch sử.

### **Quy tắc đề xuất**

| Sự kiện | onboarding\_cases | branches | members |
| ----- | ----- | ----- | ----- |
| User tự hủy hồ sơ | `CANCELLED` | `ARCHIVED` | Giữ `DU_BI` |
| Admin từ chối hồ sơ | `REJECTED` | `REJECTED` hoặc `ARCHIVED` | Giữ `DU_BI` |
| Hồ sơ hết hạn | `EXPIRED` | `ARCHIVED` | Giữ `DU_BI` |
| User nộp lại hồ sơ | Tạo `onboarding_cases` mới | Tạo branch mới hoặc tái sử dụng branch cũ theo chính sách | Có thể tái sử dụng Member dự bị |

### **Quy tắc quan trọng**

* Không dùng `deleted_at` để biểu diễn trạng thái nghiệp vụ như `REJECTED`, `CANCELLED`, `ARCHIVED`.  
* `deleted_at` chỉ nên dùng cho soft-delete mang tính kỹ thuật hoặc quản trị dữ liệu.  
* UI cây phả hệ chính thức chỉ hiển thị:  
  * `members.status = CHINH_THUC`  
  * branch đã `MERGED`  
* Branch `REJECTED`, `ARCHIVED`, `CANCELLED` chỉ xuất hiện trong màn hình quản trị hoặc lịch sử onboarding.

### **Bổ sung vào Business Process Ledger**

Các process type nên có:

```
ONBOARDING_CANCEL
ONBOARDING_REJECT
ONBOARDING_CASE_EXPIRE
ONBOARDING_BRANCH_ARCHIVE
ONBOARDING_CASE_REOPEN
```

Điểm cần tinh chỉnh: `ONBOARDING_CASE_REOPEN` chỉ dùng khi hệ thống thật sự cho phép mở lại **cùng một case**. Nếu chính sách là “nộp lại \= hồ sơ mới”, nên dùng:

```

```

và metadata lưu:

```json

  "previous_case_id": "uuid-cua-case-cu",
  "reapplication_reason": "User submitted updated family information"
}
```

## **XV.2. Khóa chỉnh sửa khi hồ sơ vào giai đoạn review hoặc merge**

Khuyến nghị B là rất quan trọng và nên đưa vào doctrine như một nguyên tắc bắt buộc.

Tuy nhiên, chỉ khóa ở Frontend là chưa đủ. Frontend có thể bị bỏ qua bằng API call trực tiếp, tab cũ, hoặc request gửi chậm.

Cần có **hai lớp khóa**:

### **Lớp 1 — UX Lock**

Khi:

```
onboarding_cases.status IN (
  SUBMITTED,
  UNDER_REVIEW,
  APPROVED,
  MERGED,
  REJECTED,
  CANCELLED
)
```

Frontend phải đưa branch sang chế độ read-only.

User không được:

* thêm Member dự bị;  
* sửa quan hệ cha/mẹ/vợ/chồng;  
* xóa Member;  
* thay đổi cấu trúc branch;  
* gửi lại hồ sơ khi chưa được yêu cầu chỉnh sửa.

Chỉ khi case ở:

```
DRAFT
PROFILE_COMPLETED
FAMILY_TREE_DRAFT
NEEDS_REVISION
```

thì user mới được chỉnh sửa.

### **Lớp 2 — Backend State Guard**

Mọi API mutation liên quan đến provisional branch phải kiểm tra:

```
onboarding_cases.status IN (
  DRAFT,
  PROFILE_COMPLETED,
  FAMILY_TREE_DRAFT,
  NEEDS_REVISION
)
```

Nếu không thỏa điều kiện, backend trả về lỗi nghiệp vụ, ví dụ:

```
ONBOARDING_CASE_NOT_EDITABLE
```

Không nên chỉ trả về lỗi chung kiểu `403 Forbidden`, vì Frontend cần biết đây là lỗi workflow để hiển thị đúng thông báo.

### **Khóa khi merge**

Khi Admin bắt đầu merge, cần chuyển case sang trạng thái trung gian:

```
MERGING
```

Luồng đề xuất:

```
UNDER_REVIEW
→ APPROVED
→ MERGING
→ MERGED
```

Lý do:

* `APPROVED` nghĩa là hồ sơ được chấp thuận về nghiệp vụ.  
* `MERGING` nghĩa là hệ thống đang thực hiện thao tác cấu trúc cây phả hệ.  
* `MERGED` nghĩa là transaction ghép cây đã hoàn tất thành công.

Nếu transaction merge thất bại:

```
MERGING → APPROVED
```

hoặc:

```
MERGING → MERGE_FAILED
```

Tôi nghiêng về phương án có trạng thái riêng:

```
MERGE_FAILED
```

vì nó giúp Admin biết hồ sơ đã được duyệt nhưng thao tác ghép cây chưa hoàn thành.

Enum `onboarding_case_status` nên mở rộng:

```

PROFILE_COMPLETED
FAMILY_TREE_DRAFT
SUBMITTED
UNDER_REVIEW
NEEDS_REVISION
APPROVED
MERGING
MERGED
MERGE_FAILED
REJECTED
CANCELLED
EXPIRED
```

## **XV.3. Composite Index cho Dashboard đa tenant**

Khuyến nghị C là đúng và nên bổ sung.

```
@@index([tenant_id, status], map: "idx_onboarding_tenant_status")
```

Đây là index cần thiết cho truy vấn phổ biến:

```
Lấy các hồ sơ onboarding đang chờ xử lý của tenant hiện tại.
```

Ví dụ:

```javascript
prisma.onboarding_cases.findMany({
  where: {
    tenant_id,
    status: {
      in: ['SUBMITTED', 'UNDER_REVIEW', 'NEEDS_REVISION']
    }
  },
  orderBy: {
    submitted_at: 'asc'
  }
});
```

Tuy nhiên, để phục vụ đúng query trên, index tốt hơn là:

```
@@index(
  [tenant_id, status, submitted_at(sort: Asc)],
  map: "idx_onboarding_tenant_status_submitted"
)
```

Index này hỗ trợ đồng thời:

4. lọc theo tenant;  
5. lọc theo trạng thái;  
6. sắp xếp danh sách hồ sơ theo thời điểm nộp.

Ngoài ra, nên có:

```
@@index(
  [user_id, status],
  map: "idx_onboarding_user_status"
)
```

Index này phục vụ State Guard và trang “Hồ sơ onboarding của tôi”.

### **Bộ index đề xuất cuối cùng**

```
@@index([tenant_id, status, submitted_at(sort: Asc)], map: "idx_onboarding_tenant_status_submitted")
@@index([user_id, status], map: "idx_onboarding_user_status")
@@index([correlation_id], map: "idx_onboarding_correlation")
@@index([reviewed_by], map: "idx_onboarding_reviewer")
@@index([primary_branch_id], map: "idx_onboarding_primary_branch")
@@index([created_at(sort: Desc)], map: "idx_onboarding_created_at")
```

Không cần tạo đồng thời cả:

```
@@index([tenant_id])
@@index([status])
@@index([tenant_id, status])
@@index([tenant_id, status, submitted_at])
```

vì sẽ gây dư thừa index. Với Dashboard, composite index ba cột thường đã bao phủ tốt nhu cầu chính.

## **XV.4. Đồng bộ enum giữa Prisma schema, Business Ledger và validator**

Khuyến nghị D là bắt buộc.

Hiện doctrine có ba lớp phải luôn đồng bộ:

| Lớp | Vai trò |
| ----- | ----- |
| PostgreSQL enum `business_process_type` | Chặn dữ liệu process type không hợp lệ ở DB |
| Prisma enum `business_process_type` | Type safety trong application |
| `business-log-schemas.js` | Chuẩn hóa và validate metadata trước khi ghi log |

Một `process_type` chỉ được xem là hoàn chỉnh khi có đủ cả ba lớp.

Ví dụ, nếu schema DB có:

```
ONBOARDING_SUBMIT
```

nhưng `BusinessLogSchemas.ONBOARDING_SUBMIT` chưa tồn tại, thì `business-logger.service.js` phải từ chối ghi log. Đây là hành vi đúng vì tránh metadata không có cấu trúc.

Ví dụ validator nên bổ sung:

```javascript
ONBOARDING_SUBMIT: (payload) => {
  if (!payload.user_id) {
    throw new Error('ONBOARDING_SUBMIT requires user_id');
  }

  if (!payload.member_id) {
    throw new Error('ONBOARDING_SUBMIT requires member_id');
  }

  if (!payload.branch_id) {
    throw new Error('ONBOARDING_SUBMIT requires branch_id');
  }

  return {
    user_id: payload.user_id,
    member_id: payload.member_id,
    branch_id: payload.branch_id,
    member_count: Number.parseInt(payload.member_count, 10) || 0,
    generation_depth: Number.parseInt(payload.generation_depth, 10) || null,
    submitted_from: payload.submitted_from || 'WEB'
  };
},
```

Tuy nhiên, theo Snapshot Doctrine hiện tại, `user_id`, `member_id`, `branch_id` không nên chỉ nằm trong `payload`.

Chúng nên xuất hiện trong `context` của `business_process_logs`:

```javascript
context: {
  target_id: onboardingCase.id,
  target_name: `Onboarding case of ${user.full_name}`,
  user_id: user.id,
  member_id: member.id,
  branch_id: branch.id
}
```

Còn `payload` nên chỉ giữ thông tin biến động hoặc dữ liệu nghiệp vụ của lần thực thi:

```javascript
payload: {
  member_count: 8,
  generation_depth: 4,
  submitted_from: 'WEB'
}
```

Điều này giúp metadata nhất quán:

```json
{
  "context": {
    "target_id": "onboarding-case-uuid",
    "target_name": "Onboarding case of Nguyễn Văn A",
    "user_id": "user-uuid",
    "member_id": "member-uuid",
    "branch_id": "branch-uuid"
  },
  "payload": {
    "member_count": 8,
    "generation_depth": 4,
    "submitted_from": "WEB"
  }
}
```

# **Kết luận review**

Bốn phản biện đều có giá trị và nên được đưa vào tài liệu. Nhưng phiên bản chính thức nên bổ sung thêm ba quyết định kiến trúc:

4. Có trạng thái `MERGING` và `MERGE_FAILED` để tách “được duyệt” khỏi “đã ghép cây thành công”.  
5. State Guard phải được thực thi ở cả Frontend và Backend.  
6. Các ID định danh chính của đối tượng phải được đặt trong `context` snapshot; `payload` chỉ chứa dữ liệu đặc thù của lần thực thi.

Như vậy doctrine Onboarding sẽ đủ vững cho Phase 6B, đồng thời không bị giới hạn khi sau này triển khai merge phức tạp, retry, review nhiều vòng hoặc workflow cho nhiều cấp trưởng chi.
